# 基于 MongoDB 的问卷系统核心文档

## 1. 系统说明

### 1.1 项目目标
本系统用于完成问卷的创建、发布、填写与统计分析，支持登录用户管理自己的问卷，并支持通过分享链接对外收集问卷回答。

### 1.2 技术栈
- 后端：Node.js + Express
- 数据库：MongoDB + Mongoose
- 认证：JWT（Bearer Token）
- 前端：原生 HTML + CSS + JavaScript（由 Express 静态托管）
- 测试：Jest + Supertest

### 1.3 模块划分
- 用户认证模块：注册、登录、签发与校验 Token。
- 问卷管理模块：创建问卷、编辑信息、发布、关闭、删除。
- 题目管理模块：新增题目、更新题目、删除题目、题目排序。
- 填写提交模块：通过分享码获取问卷，提交答案并完成校验。
- 统计分析模块：整卷统计与单题统计。

### 1.4 运行与访问
- 服务默认端口：3000。
- 前端入口：浏览器访问根路径后返回 index 页面。
- API 根路径：/api。

### 1.5 业务规则概览
- 问卷状态：draft（草稿）/ published（已发布）/ closed（已关闭）。
- 题目管理仅允许在草稿状态进行。
- 问卷发布前至少要有 1 道题目。
- 非匿名问卷必须登录才能填写。
- 问卷过截止时间后禁止提交。

## 2. 数据库设计

系统包含 4 个核心集合：users、surveys、questions、responses。

### 2.1 users（用户）
用途：存储账号信息与密码摘要。

关键字段：
- _id：ObjectId，主键。
- username：String，唯一，2-30 字符。
- password：String，保存前由 bcrypt 加密。
- createdAt：Date，创建时间。

约束与特性：
- username 唯一索引。
- 序列化输出时隐藏 password。

### 2.2 surveys（问卷）
用途：存储问卷元信息与发布状态。

关键字段：
- _id：ObjectId，主键。
- creatorId：ObjectId，关联 users._id。
- title：String，问卷标题。
- description：String，问卷描述。
- allowAnonymous：Boolean，是否允许匿名。
- status：String，枚举 draft/published/closed。
- deadline：Date，截止时间，可为空。
- shareCode：String，分享码，唯一。
- createdAt / updatedAt：Date，时间戳。

索引：
- creatorId 索引，用于“我的问卷”查询。
- shareCode 唯一索引，用于公开填写入口定位。

### 2.3 questions（题目）
用途：存储问卷题目、校验规则与跳转规则。

关键字段：
- _id：ObjectId，主键。
- surveyId：ObjectId，关联 surveys._id。
- order：Number，题目顺序。
- type：String，枚举 single_choice/multiple_choice/text_input/number_input。
- title：String，题干。
- required：Boolean，是否必答。
- options：String[]，选择题选项。
- validation：Object，按题型配置校验参数。
- jumpRules：Object[]，条件跳转规则。

validation 子字段：
- 多选：minSelect、maxSelect、exactSelect。
- 文本：minLength、maxLength。
- 数字：min、max、integerOnly。

jumpRules 子结构：
- condition.type：equals/contains/gt/lt/gte/lte。
- condition.value：比较目标值。
- targetQuestionOrder：命中后跳转题号。

索引：
- 复合索引 surveyId + order，用于按题序快速读取。

### 2.4 responses（回答）
用途：存储每次提交的问卷答案。

关键字段：
- _id：ObjectId，主键。
- surveyId：ObjectId，关联 surveys._id。
- respondentId：ObjectId 或 null，匿名时为 null。
- answers：Object[]，按题保存回答。
- submittedAt：Date，提交时间。

answers 子结构：
- questionId：ObjectId，关联 questions._id。
- questionOrder：Number，提交时的题序。
- value：Mixed，可为 String、Number、String[]。

关系说明：
- 一个 survey 对应多个 question。
- 一个 survey 对应多个 response。
- 一个 response 包含该次作答的 answers 列表。

## 3. API 说明

### 3.1 认证约定
- 认证头：Authorization: Bearer <token>
- 需要登录的接口必须携带 Token。
- 部分接口使用可选认证：带 Token 则识别用户，不带也可访问（例如匿名问卷填写页读取）。

### 3.2 用户认证接口

1. POST /api/auth/register
- 说明：注册账号。
- 请求体：username, password。
- 成功：201，返回 user 与 token。
- 常见失败：400 参数非法，409 用户名已存在。

2. POST /api/auth/login
- 说明：用户登录。
- 请求体：username, password。
- 成功：200，返回 user 与 token。
- 常见失败：400 参数缺失，401 用户名或密码错误。

### 3.3 问卷管理接口（需登录）

1. POST /api/surveys
- 说明：创建问卷。
- 请求体：title, description, allowAnonymous, deadline。

2. GET /api/surveys
- 说明：获取当前用户创建的问卷列表。

3. GET /api/surveys/:id
- 说明：获取单个问卷详情（仅创建者可见）。

4. PUT /api/surveys/:id
- 说明：更新问卷基本信息，仅 draft 状态可修改。

5. PUT /api/surveys/:id/publish
- 说明：发布问卷。
- 规则：必须是 draft 且至少有一题。

6. PUT /api/surveys/:id/close
- 说明：关闭问卷。
- 规则：必须是 published。

7. DELETE /api/surveys/:id
- 说明：删除问卷，同时级联删除题目与回答。

常见失败状态：
- 401 未登录。
- 403 非创建者无权限。
- 404 问卷不存在。
- 400 业务状态不允许。

### 3.4 题目管理接口（需登录）

1. POST /api/surveys/:surveyId/questions
- 说明：新增题目。
- 支持字段：type, title, required, options, validation, jumpRules, order。
- 若未传 order，系统自动追加到末尾。

2. GET /api/surveys/:surveyId/questions
- 说明：获取问卷全部题目（按 order 升序）。

3. PUT /api/questions/:id
- 说明：更新题目。
- 限制：仅 draft 状态可改。

4. DELETE /api/questions/:id
- 说明：删除题目。
- 限制：仅 draft 状态可删。
- 额外处理：清理其他题目中指向该题 order 的跳转规则。

### 3.5 填写与提交接口

1. GET /api/survey/:shareCode
- 说明：通过分享码获取问卷与题目。
- 认证：可选认证。
- 规则：问卷必须 published 且未过期。
- 非匿名问卷在未登录时返回 401。

2. POST /api/survey/:shareCode/submit
- 说明：提交问卷答案。
- 认证：可选认证（非匿名问卷实际要求登录）。
- 请求体：answers 数组，元素包含 questionId 与 value。
- 成功：201，返回 responseId。

常见失败状态：
- 400 问卷状态不允许、已截止、答案格式或答案校验失败。
- 401 非匿名问卷但未登录。
- 404 分享码不存在。

### 3.6 统计接口（需登录，且仅创建者）

1. GET /api/surveys/:id/stats
- 说明：整卷统计。
- 返回：总提交数 + 每题统计。

2. GET /api/surveys/:id/stats/:questionId
- 说明：单题统计。

统计内容按题型输出：
- single_choice：各选项计数。
- multiple_choice：各选项累计被选次数。
- text_input：答案列表。
- number_input：答案列表 + 平均值 + 最小值 + 最大值。

## 4. 关键逻辑说明

### 4.1 JWT 鉴权与可选鉴权
- 强鉴权中间件用于后台管理接口：必须携带合法 Bearer Token。
- 可选鉴权用于公开填写接口：有 Token 则识别身份，无 Token 也可继续；是否允许匿名由问卷配置决定。

### 4.2 答案校验引擎
提交时服务端对每道可达题目执行统一校验：
- 必答校验：空值或空数组在 required 题目上直接失败。
- 单选校验：答案必须存在于 options。
- 多选校验：值必须是数组，且选项合法，数量满足 min/max/exact 规则。
- 文本校验：类型必须为字符串，长度满足 minLength/maxLength。
- 数字校验：可转数字，满足 integerOnly、min、max。

该校验在后端执行，确保绕过前端也无法提交非法数据。

### 4.3 跳转逻辑引擎
系统基于 jumpRules 决定下一题，流程为：
1. 当前题无规则，走默认下一题。
2. 当前题未作答（含空数组），不触发跳转，走默认下一题。
3. 依次匹配规则 condition，命中则跳到 targetQuestionOrder。
4. 若未命中，走默认下一题。

支持条件：equals、contains、gt、lt、gte、lte。
其中 equals/contains 做了字符串层统一比较，数字比较做了有限值保护，避免类型差异与 NaN 导致误判。

### 4.4 提交流程与防环策略
提交接口不会盲信前端“显示了哪些题”，而是服务端按题序与跳转规则重新模拟作答路径：
- 先把提交 answers 转为 questionId 到 value 的映射。
- 从首题开始，逐题校验并决定下一题。
- 只保存路径上有效且非空的答案。

为避免跳转规则形成死循环导致接口卡死，加入两层保护：
- visitedIdx：同一题索引再次访问则中断。
- maxSteps：总步数超过阈值则中断。

### 4.5 统计聚合逻辑
统计模块先按 surveyId 取出所有回答，再按题目逐一计算：
- 从每份 response.answers 中提取对应 questionId 的值。
- 按题型分支聚合输出，保证统计结构稳定。

### 4.6 权限与数据安全策略
- 所有管理端数据（问卷、题目、统计）均按 creatorId 做拥有者校验。
- 删除问卷时执行级联删除，避免孤儿题目与回答。
- 删除题目时同步清理其他题目的失效跳转规则，保证逻辑一致性。

---

## 5. 小结

本系统以“后端规则权威化”为核心设计原则：
- 前端负责交互体验；
- 后端负责权限、状态、校验、跳转与统计。

该设计能保证在复杂问卷场景下仍具备较高的一致性、可维护性与可扩展性。
