# AI 使用过程说明

## 通过与 AI 交流形成数据库 Schema 的过程
在数据库设计阶段，采用“提出约束-反问理由-对照代码落地”的方式推进。

我和 AI 的交流重点分为 4 轮：
1. 先定边界：明确业务是问卷全流程（注册、建卷、题目、填写、统计），并强调 MongoDB 是必选，且第二阶段要可扩展，不能推倒重来。
2. 再定实体：我要求 AI 先回答“为什么是 users、surveys、questions、responses 这 4 个集合”，而不是先写字段。通过这一步我确认了聚合根：问卷是管理单元，题目和回答是高增长子数据，应该解耦存储。
3. 对关键分歧做技术取舍：我重点追问了“题目到底嵌入 survey 还是独立集合”。最终选择独立 questions，因为我判断了 3 个风险：文档膨胀、题目级查询困难、二阶段变更成本高。
4. 最后做可运行校验：我要求每个 schema 决策都映射到实际接口和查询路径，确保不是纸面设计。例如 shareCode 必须唯一索引，因为它是公开填写入口；questions 需要 surveyId + order 复合索引，因为读取路径是按问卷顺序取题。

### 交流示例（节选）
一：关于 questions 是否嵌入 surveys
1. 我的问题：如果把题目嵌入问卷文档，开发会不会更简单？
2. AI 回答要点：嵌入读问卷快，但题量增加会导致文档变大，题目级统计和局部更新复杂。
3. 我的判断：本项目需要题目增删改和跳转规则维护，且二阶段可能新增题目属性，独立集合更稳妥。
4. 最终决策：questions 独立集合，survey 与 question 通过 surveyId 关联。

二：关于 responses 的存储边界
1. 我的问题：回答是否可以直接挂在 surveys 下面，减少一次关联？
2. AI 回答要点：短期可行，但提交量上来后会形成写入热点，统计查询也不利于扩展。
3. 我的判断：填写是高频写入路径，必须与问卷主文档解耦；统计也需要按 surveyId 高效聚合。
4. 最终决策：responses 独立集合，并建立 surveyId 索引。

三：关于索引是否“全建”
1. 我的问题：是不是把常用字段都建索引最保险？
2. AI 回答要点：索引越多写入成本越高，应按访问路径建立必要索引。
3. 我的判断：本项目高频路径是“我的问卷”“分享码访问”“按题序读取”“按问卷统计”。
4. 最终决策：建立 creatorId、shareCode、(surveyId, order)、responses.surveyId 四类核心索引，不盲目加索引。

四：关于 jumpRules 的字段类型
1. 我的问题：condition.value 用固定类型（全字符串）是否更容易实现？
2. AI 回答要点：实现简单但会损失数字比较语义，后续 gt/lt 容易出现类型转换问题。
3. 我的判断：跳转条件天然是混合类型场景，应该保留表达能力并在逻辑层做统一比较。
4. 最终决策：condition.value 使用 Mixed，服务端在匹配时做字符串与数字场景分流处理。
